## 五、仲裁协议

### 自动触发词（响应中含以下词汇时自动进入仲裁）

- "规范冲突" / "规则矛盾" / "不确定应该" / "两种方式都合理"
- "这与架构不符" / "需求超出范围" / "违反现有规范"

### 手动强制触发

```text
[ARBITRATE] 冲突描述
当前上下文：[正在执行的任务]
冲突双方：[主张A] vs [主张B / 规则X]
影响范围：[受影响的文件/功能]
```

### 冲突分类与处理

| 类型 | 条件 | 处理方式 |
| --- | --- | --- |
| **A 型**（规则明确） | 本文件有明确规定 | 自动裁定，引用规则原文，即时生效 |
| **B 型**（规则模糊） | 规范未明确覆盖，两种方式各有依据 | 协商裁定，领域权威加权投票，登记规范改进待办 |
| **C 型**（超出权限） | 满足任意一条升级条件 | **必须人工介入，任务暂停** |

**C 型升级条件（满足任意一条）**：

- 影响超过 3 个功能模块的架构变更
- 删除现有功能或弃用现有 API
- 引入新的技术栈或第三方依赖
- A/B 型裁定后仍有争议

### 裁定记录格式

仲裁结束后追加到 `.github/memory/arbitration-log.md`：

```markdown
### [ARB-YYYY-序号] 裁定标题

- **时间戳**：YYYY-MM-DD HH:MM
- **冲突类型**：A型 / B型 / C型
- **冲突描述**：[一句话]
- **裁定结论**：[具体决策]
- **受影响文件**：[需要更新的文件]
- **状态**：已生效 / 待人工确认
```

## 九、测试规范

### 测试类型

| 类型 | 覆盖目标 | 优先级 |
| --- | --- | --- |
| 单元测试 | UseCase、纯逻辑类 | 最高 |
| 组件测试 | 独立组件交互 | 中 |
| 集成测试 | 完整用户流程 | 低 |

### UseCase 必测场景（缺一不可）

- 正常路径：返回预期数据
- 业务校验失败：抛出带用户可读信息的异常
- 响应数据为空/null：抛出异常
- 依赖层异常（网络/存储）：向上传播

### ViewModel 必测场景（缺一不可）

- 初始状态是 `initial`
- 调用方法后进入 `loading`
- 成功后转为 `success`
- 失败后转为 `error`，`errorMessage` 无技术前缀
- `reset()` 后恢复 `initial`

### 覆盖率目标

```text
[FILL]
UseCase 层：建议 100%
ViewModel 层：建议 80%
Repository 层：建议 70%
```

## 十二、紧急覆盖协议

以下情况允许人工直接覆盖 Agent 决策，无需仲裁流程：

1. Agent 产生的代码导致应用崩溃或数据丢失
2. 安全漏洞（OWASP Top 10 相关）
3. 性能回归超过 30%

紧急覆盖后，**必须**在 `.github/memory/decisions.md` 中记录覆盖原因。
